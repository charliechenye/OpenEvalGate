# Validation Matrix

Use this table to choose the smallest validation that proves a change. `.github/workflows/ci.yml` remains the source of truth for required CI behavior.

## Principles

- Run focused checks during iteration.
- Run the full suite once after focused checks pass when executable behavior changed.
- Do not repeatedly run full validation after every edit.
- Use `-q` for successful focused tests.
- Use `-vv`, full durations, or full logs only for diagnosis.
- Do not run Python tests for a pure prose-only change unless executable snippets, paths, packaging metadata, or workflow behavior changed.

Focused-test convention:

```bash
python -m pytest tests/<relevant_test_file>.py -q
```

Broader Python validation from CI:

```bash
python -m pytest \
  -n 4 \
  --dist=load \
  --maxschedchunk=1 \
  --durations=20 \
  --durations-min=1.0
```

Other common checks:

```bash
python -m compileall -q openevalgate
python -m build
git diff --check
```

## Matrix

| Change type | Focused iteration checks | Required completion checks | Canonical/generated artifact handling |
| --- | --- | --- | --- |
| Prose-only documentation | `git diff --check`; inspect rendered links/paths when changed | `git diff --check` and file existence/link checks relevant to the edit | None; do not read or regenerate canonical reports. |
| Executable documentation or command examples | Run the documented command or the narrowest equivalent smoke check | Relevant focused tests plus `git diff --check`; run broader Python validation if executable behavior changed | Regenerate only if command output intentionally changes report bytes. |
| Schema/parser/validator changes | `python -m pytest tests/test_schema.py -q` or `tests/test_validator.py -q` | CI pytest command once, `python -m compileall -q openevalgate`, `git diff --check` | Regenerate affected reports when accepted inputs or validation summaries change. |
| Eval-result integrity changes | `python -m pytest tests/test_eval_results.py -q` | CI pytest command once, `python -m compileall -q openevalgate`, `git diff --check` | Reproduce affected example reports if summaries, blockers, or report wording change. |
| Review-policy or behavioral-sufficiency changes | `python -m pytest tests/test_review_policy.py tests/test_assessment.py -q` | CI pytest command once, `python -m compileall -q openevalgate`, `git diff --check` | Reproduce affected controlled/shadow examples; all examples if shared semantics changed. |
| Blocker, scoring, or assessment changes | `python -m pytest tests/test_hard_gate_policy.py tests/test_assessment.py -q` | CI pytest command once, `python -m compileall -q openevalgate`, `git diff --check` | Reproduce affected canonical reports; all examples for shared blocker, score, or recommendation semantics. |
| Report rendering changes | `python -m pytest tests/test_report.py -q` | CI pytest command once, `python -m compileall -q openevalgate`, `git diff --check` | Never hand-edit reports; regenerate affected reports through the CLI and compare bytes. |
| CLI changes | Select the relevant CLI coverage from `tests/test_report.py`, `tests/test_project_inspection.py`, and `tests/test_version.py`; run only the files or test nodes affected by the command path | CI pytest command once, `python -m compileall -q openevalgate`, relevant canonical CLI smoke checks, `git diff --check` | Reproduce reports if CLI output or report-writing semantics changed. |
| Canonical example input changes | Run `openevalgate validate`, `openevalgate check`, and `openevalgate report --output` for the affected example | Focused tests for the changed behavior plus `git diff --check`; broader Python validation if executable behavior changed | Regenerate that example report and compare bytes; all examples only for shared report semantics. |
| Packaging or metadata changes | Relevant `tests/test_distribution_artifacts.py` or `tests/test_version.py` focused tests | `python -m build`, artifact inspection, clean-wheel install, installed CLI smoke testing consistent with CI, `git diff --check` | Reproduce reports only if installed CLI smoke output or report semantics changed. |
| CI workflow changes | Static review of the edited workflow step and any command it invokes | The workflow run is primary validation; run local command checks only for changed commands that are practical locally | None unless workflow changes canonical generation or comparison behavior. |
| Dependency changes | Focused tests for importing and using the dependency | CI pytest command once, `python -m compileall -q openevalgate`, `python -m build`, installed smoke checks when packaging is affected, `git diff --check` | Regenerate affected reports only if dependency changes deterministic output. |

## Canonical Rules

- Never hand-edit canonical reports.
- Regenerate affected reports through the CLI.
- Compare generated bytes against checked-in canonical files.
- If shared report behavior changed, reproduce all canonical reports.
- Packaging changes require build, artifact inspection, clean-wheel install, and installed CLI smoke testing consistent with CI.
- CI-only changes are primarily validated by the workflow run, not by repeatedly running unrelated local tests.

## Escalation Rule

When uncertain, explain why broader validation is necessary before running it. Do not default automatically to every command.
